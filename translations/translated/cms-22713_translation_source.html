Vantagens e Desvantagens dos Frameworks para WordPress

O conceito de um framework é ser algo bastante comum tanto na web atual quanto dentro dos ambientes de desenvolvimento de software. Para o frontend, temos frameworks como o Bootstrap e o Foundation; para aplicações web temos frameworks como o Yii e o Rails; e para o desktop (e té mesmo algumas aplicações web) temos frameworks como o .NET. E esses são os nomes de alguns poucos.

De forma geral, frameworks são bons: Eles proveem um nível de abstração com o qual nós, enquanto desenvolvedores, podemos trabalhar e que facilitam a criação de funcionalidades genéricas — como login de usuários e autenticação — assim como códigos mais específicos, como a lógica própria para o problema que o seu software pretende resolver.

Em resumo, a Wikipedia define um framework da seguinte forma:

Um framework de software é um ambiente de software universal e reusável que provê certas funcionalidades como parte de uma plataforma de software maior, para facilitar o desenvolvimento de aplicações computacionais, produtos e soluções. Frameworks de software podem incluir programas de apoio, compiladores, bibliotecas de código, conjuntos de ferramentas e interfaces de programação de aplicações (APIs) que provem todo um conjunto diferente de componentes que permitem o desenvolvimento de um projeto ou solução.

Assim como várias plataformas modernas, não é diferente com o WordPress. Claro, o WordPress em si não é uma um framework - é um aplicativo que pode ser estendido através do uso de sua várias APIs. Mas há um número diferente de frameworks que estão disponíveis e todas tem como objetivo facilitar a escrita de código repetitivo.

Mas isso é uma coisa boa? Isso é, há sentido em ter tantas opções de frameworks a se escolher, uma vez que elas se assemelham à construção de temas, plugins ou aplicações web baseada na plataforma?

Geralmente, depende daquilo que você quer fazer.

No Tuts+, tentamos cobrir o máximo de frameworks possível - algumas para o WordPress, outras para várias linguagens de programação e plataformas que foram mencionada anteriormente - mas, neste artigo, não falaremos de um framework específica.

Ao invés disso, veremos as vantagens e desvantagens em selecionar um framework para se trabalhar com o WordPress, para ajudar na tomada das melhores decisões na hora de construção sites, aplicações e outros projetos usando o WordPress como base.

Com o WordPress, é possível misturar vários frameworks para alcançar um resultado, e esse artigo assume que você esteja avaliando o uso de um único framework para ajudar em seu trabalho, ou não.

Comparando Frameworks

Em relação ao WordPress, atualmente (e geralmente) há dois tipos de frameworks:

  1. Frameworks de Arrastar e soltar
  2. Frameworks de Opções

Geralmente, os frameworks de arrastar e soltar são usados para a construção da interface de usuário do site. Inúmeras vezes, elas vem na forma de diversas páginas administrastivas no Painel do WordPress, permitindo que você ajustes elementos pré-definidos em uma página, com o resultado final gerado na ordem criada para o usuário final ver.

Por outro lado, os frameworks de opções usão funcionalidade através de códigos e costumam prover vários tipos de abstrações em cima de uma ou mais APIs do WordPress. Isso inclui, mas não se limita a, API de Configurações (Settings API), API de Meta Dado (para os Posts, Usuários, Comentários e por aí vai), e até algumas APIs relacionadas a modelos.

Qualquer que seja o caso, há vantagens e desvantagens para cada tipo de framework, as quais são importantes considerar ao escolher se você trabalhar com uma dada framework ou não.

Vantagens

Desenvolvimento Rápido

Uma das maiores vantagens em usar um framework é a velocidade que você é capaz de alcançar em gerar funcionalidades. 

Algumas vezes, as pessoas referem-se a isso como prototipação rápida, mas os frameworks WordPress amadureceram tanto que, geralmente, você é capaz de criar interfaces, sites e aplicações complets — não apenas protótipos — ao trabalhar com a API oferecida.

Embora a construção de sites focados em conteúdo com WordPress é bem simples e direto, mesmo sem quaisquer frameworks, muitos deles — especialmente as de arrastar e soltar — tornam essa tarefa muito mais rápida ao criar layouts mais elaborados para as páginas.

Abstração

Para os frameworks que lidam com APIs do WordPress, eles podem prover um bom nível de abstração, como o código repetitivo que estamos acostumados a escrever já vem embutido no framework.

Talvez um dos melhores exemplos estejam nos frameworks para a API de Configuração. A API de Configuração, embora poderosa, também é muito verbosa e incômoda quando precisamos buscar os elementos para apresentar na tela. Os frameworks tentar resolver esse problema probendo uma maneira mais fácil de criar configurações e opções com o mínimo de chamadas possível e um código mais limpo.

Claro, abstrações não existem apenas para a API de Configuração. Escolher usar um desses frameworks pode economizar muito do seu tempo e ajudá-lo nas tarefas tediosas que precisam de muito código repetitivo no WordPress.

Representação Visual

Outra vantagem em usar frameworks é que eles ajudam a diminuir a lacuna entre o que esperamos que aconteça e o que realmente acontecerá. 

Percebemos isso, principalmente, nos frameworks de arrastar e soltar: Você tem conjuntos prédefinidos de regiões com as quais se deve trabalhar, como cabeçalhos, barras laterais, áreas de conteúdos e rodapés. Então, você posiciona as regiões dentro de um recipiente (container). Uma vez feito isso, tais regiões aparecerão no frontend, exatamente, como você as arranjou no backend.

O mesmo pode ser dito para frameworks que trabalham com APIs. Embora a natureza visual da escrita de código não seja tão clara quanto trabalhar com um editor de modelos, um bom frameworks pode prover classes, funções e outras facilidades que ajudam a entender mais fácil como as coisas trabalham em conjunto.

Um exmeplo disso é através da criação de uma configuração no Painel do WordPress. Digamos, por exemplo, que você quer adicionar um campo textarea que só aceite texto simples (isso é, sem código HTML ou scripts), e você precisa rotulá-lo, verificar seus dados e mostrar o resultado.

Usando um framework, aquilo que poderia levar três chamadas a funções diferentes e duas funções callback, pode ser feito com uma única chamada a uma única função e, talvez, uma ou duas funções callback. Além disso, os nomes das funções são mais simples e claros, indicando o que elas realmente fazem e com que tipo de elementos elas trabalham.

Desvantagens

Administração de Dependência

Por outro lado, se escolher usar um framework, automaticamente, criará uma dependência entre você, código de terceiros e o WordPress.

Isso significa que seu código é baseado em um framework e o framework é baseado no WordPress, assim, as capacidades do framework são limitadas pela versão mais recente do WordPress que ele dá suporte.

Se o framework mantém-se atualizado com o WordPress enquanto está em desenvolvimento, e o framework tem lançamentos liberados junto com atualizações maiores do WordPress, será fácil manter seu código em sincronia com as versões mais recentes.

Por outro lado, se o framework não se atualiza para a versão mais recente do WordPress, seu código só pode estar tão atualizado (seguro e garantido) quanto a versão mais recente do WordPress suportada pelo framework.

Deprecação

Outro desafio que existe ao usar frameworks é que, muita vezes, os frameworks são envóluvros elaborados para funcionalidade já existente dentro do próprio WordPress. Por conta disso, terá vezes que os frameworks proverão acessos a funções que talvez não estejam disponívels no WordPress após alguma atualização.

Digamos, por exemplo, que você usa um framework que provê acesso a certas APIs que se tornam deprecadas na versão mais recente do WordPress. Além do mais, o framework não depreca seu código até que seja lançada uma nova versão.

Isso significa que o código criado com base no código da framework está invocando funções deprecadas. Por um ou dois lançamentos, isso pode nao ser problema, mas, uma vez que a funcionalidade já não existe mais no WordPress e o framework ainda não resolveu isso, talvez você não possa usar a versão mais recente do WordPress ou algumas das novas funcionalidades, uma vez que o framework não deprecou corretamente seu código em relação ao código do WordPress.

Limitação de Funcionalidades

Os frameworks pode sem bem poderosos, mas ao usá-los, algumas vezes é uma boa ideia usá-la por completo. Isso é, ao usar um framework, é bom criar todo seu código usando o framework como base, ao invés de apenas uma parte do código usar o framework e outra parte usar as APIs do WordPress diretamente.

Em outras palavras, você não quer misturar códigos, com uma parte usando o framework que você conhece e outra parte que não use o framework.

No fim das contas, o propósito de um framework, como definido anteriormente, é prover uma abstração para funcionalidades genéricas.

Por isso, se um framework está incompleta ou não tem alguma funcionalidades específica que o WordPress oferece, você tem duas opções a tomar:

  1. Renunciar aquela funcionalidade e esperar o framework disponibilizá-la,
  2. Deixar o framework de lado e usar o código base do WordPress junto do seu código baseado no framework para alcançar o resultado desejado.

Inicialmente, parece claro que, se o framework não oferece algo que você precisa, basta usar o que o WordPress oferece. Mas, ao fazer isso, vocês estará misturando tipos de código (do framework e do WordPress), gerando um nível a mais de complexidade para a manutenção do software.

Se depois o framework introduz um conjunto de funcionalidades que cobre o código que você escreveu usando o código base do WordPress, você terá a tarefa de atualizar tal código para adaptar-se ao framework. A partir daí, se o framework não se mantiver atualizado com o WordPress, você voltará a ter os problemas de deprecação e administração de dependências.

Conclusão

Não é fácil listar todas as vantagens e desvantagens ao usar um framework, muito menos escolher se é preciso usar um, mas os pontos mencionados acima são algumas das considerações mais comuns que alguém que trabalha com WordPress frameworks deveriam ter em mente, antes de selecionar um framework.

Além do mais, como mencionado anteriormente, este artigo não foi escrito para servir como argumento a favor ou contra o uso de frameworks para WordPress. Ao invés disso, foi escrito para apresentar tanto as vantagens e desvantagens do uso de frameworks, para ajudá-lo a tomar uma melhor decisão.

Como qualquer coisa no mundo do desenvolvimento, sempre haverá alguma troca. Sempre será algo relacionado ao que você quer trocar em favor de alguma funcionalidade, dependência ou estabilidade, ao mesmo tempo que pesa o software base que sempre está sob desenvolvimento e possuirá lançamentos de novas versões a cada seis meses.

Dependendo da natureza do seu trabalho, um framework pode ser uma das formas de dar sequência; para outros, nem tanto. Qualquer que seja o caso, tenha certeza de averiguar tudo que for possível na hora de decidir em usar, ou não um framework em seu projeto atual ou em projetos futuros.
