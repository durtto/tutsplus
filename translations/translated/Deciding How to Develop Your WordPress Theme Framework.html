Decida Como Desenvolver Sua framework Para Temas de WordPress

Na primeira parte dessa série, mostrei como as *frameworks* para temas de WordPress funcionam, bem como os diferentes tipos de *frameworks* que existem.

Antes de você começar a construir sua própria framework, você precisará considerar como ela funcionará e como poderá ser usada, para que você possa lançar mão da melhor abordagem de desenvolvimento possível.

Nesse tutorial, apresentarei os fatores a se considerar nessa hora, como: se a framework será pública ou privada; se ela poderá ser usada por pessoas comuns ou só desenvolvedores; e quais características adicionais você pode querer incluir.

Há dois momentos durante a decisão da abordagem: identificar como sua framework será usada e, baseado nisso, identificar o que precisa ser incluído nela.

Como Seu framework Para Temas Será Usado?

A forma como sua framework será usada influenciará aquilo que você incluirá e como a estruturará.

Considere o seguinte:

  • Seu framework para temas será só para você ou outros desenvolvedores terão acesso?
  • Sua framework será usada só por desenvolvedores e/ou por usuários sem experiência em programação?
  • Sua framework será pública ou privada?

Só Para Você ou Para os Outros?

Se sua framework é de uso pessoal, você precisará pensar apenas em suas necessidades ao desenvolvê-la; agora, faz todo o sentido estar preparado para o futuro e torná-la o mais robusta possível. Assim, você deveria:

  • Usar os Padrões de Codificação do WordPress
  • Adotar o princício DRY (do not repeat yourself - não se repita)
  • Usar códigos que possam ser validados pelo validador da W3C. Também deve garantir que seu código é acessível
  • Inclua comentários - mesmo que ninguém mais vá olhar para seu código, você se surpreenderá o quão fácil é esquecer algum trecho de código faz, quando precisar modificá-lo daqui alguns meses
  • Use controle de versão para as atualizações de sua framework

Caso sua framework tenha um público alvo maior que apenas você, como seus colegas ou outros desenvolvedores, é preciso que adote todas as práticas mencionadas acima e considere as listadas abaixo:

  • Prover documentação com um resumo da estrutura, funções e ganchos (*hooks*) da sua framework
  • As formas de compartilhamento e colaboração - usar um sistema de colaboradores como o GitHub facilitará bastante
  • Documentar versões ou adicioçã de links para metas alcançadas.

Para Desenvolvedores ou Usuários?

Algumas *frameworks* são voltadas para usuários finais que querem ter o poder da customização, sem precisar digitar um única linha de código. Outras, por outro lado, são voltadas apenas para desenvolvedores, provendo ganchos e funções que outros podem usar para customizar e estender. Outras são apropriadas para ambos, com ótimas interface de usuário e API.

Só porque sua framework será usada por não desenvolvedores, não quer dizer que ela será lançada ao público - talvez tenha companheiros de trabalho que são designers, para os quais quer dar acesso, ou talvez queira permitir que seus clientes a usem para customizar seus sites.

Se sua framework for para não programadores, talvez queira incluir:

  • Uma ou mais telas de opções do tema, nas quais os usuários poderão realizar as customizações
  • Acesso ao customizador de tema, a qual talvez dê preferência, ao invés de criar telas de opções para o tema, permitindo os usuários verem as mudanças na hora. Talvez escolha ambas as opções
  • Áreas de widgets que permitam usuários adicionar seus próprios conteúdos em uma varieade de lugares na página
  • Menus para que seus usuários possam navegar pelo site (talvez queira adicionar mais de uma área para menus)
  • Suporte a temas filhos para que os usuários possam instalar e criar sites rapidamente
  • Bibliotecas para quaisquer funcionalidades que queira incluir, como sliders e light boxes
  • Documentação e suporte para que os usuários saibam como usar seu trabalho (é útil, mas não deixe que tome todo o seu tempo)

Se seu público alvo é de desenvolvedores que usarão sua framework junto dos temas filhos e plugins deles, talvez queira considerar alguns dos itens acima, mas, também, inclua algumas das características da lista abaixo:

  • Ganchos de Ações (*action hooks*), permitindo que os desenvolvedores insiram seus próprios códigos no tema sem precisar duplicar algum arquivo já existente no tema
  • Ganchos de filtro (*filter hooks*), permitindo que os desenvolvedores possam alterar a saída dos arquivos do tema
  • Funções customizadas que os desenvolvedores possam lançar mão em seus temas filho
  • Partes de modelo do tema e arquivos de inclusão para redução de duplicação de código. Isso é algo que você mesmo se beneficiará, uma vez que comece a trabalhar *com* a framework e que os outros desenvolvedores achem útil se, na teoria, precisarem criar novas partes de modelo para seus temas filho. Tenha certeza de seus arquivos estarem nomeados corretamente e bem estruturados, adicione comentários dizendo quais arquivos os invocam, para facilitar a busca.

Público ou Privado?

Se você planeja liberar sua framework para o público, há toda uma nova gama de considerações:

  • Se submeter sua framework como um tema através do Repositórios de temas do WordPress, você precisará aderir ao Guia de Revisão de Temas
  • Como os usuários usarão sua framework em diversos cenários e tipos de sites, é preciso que teste sua framework nos mais variados contextos e, quem sabe, pedir a ajudar de outros usuários e desenvolvedores para um teste mais apurado
  • Alguma documentação é essencial, seja para os desenvolvedores, seja para os usuários finais, dependendo do seu público alvo

Você também precisará considerar como será o plano de marketing para a framework: mesmo que seja gratuita, você quer que as pessoas a usem, então é preciso que haja publicidade, seja via site, mídia social, SEO, lojas terceirizadas de temas, boca-a-boca, encontros locais, WordCamps ou outros métodos.

O Que Sua framework de Tema Deveria Incluir?

Uma grande parte das funcionalidades do seu tema será determinada pelas necessidades dos usuários que você acabou de selecionar. Uma vez decidido quem será o público da framework, e após perguntar quais as necessidades dele, crie uma lista de funcionalidades que seu tema incluirá.

A lista abixo apresenta (mas não está limitada a) alguns pontos:

  • Arquivos de modelo (inclusão de partes do modelo e outros)
  • Gunções
  • Ganchos de Ação e Filtro
  • Áreas de Widget
  • Menus
  • Telas de opções e configurações
  • Suporte ao Customizador de Temas
  • Documentação
  • Temas filhos

Para cada um, identifique:

  • Qual é a característica
  • O que ela fará
  • Onde o código dela estará

Da mesma forma que as funcionalidades decididas a partir dos diferentes grupos de usuários, talvez queira incluir outras funcionalidades como:

  • Sua framework terá modelos integrados ou será configurável a partir de temas filhos?
  • O que tanto será incluído no tema pai? Algumas *frameworks* usam estilos CSS mínimos, enquanto outros (como a nossa) são criadas usando CSS Orientado a Objetos (OOCSS) para facilitar a estilização de temas filhos.
  • Sua framework será responsiva ou isso será por conta dos temas filho? Se seu tema pai é responsivo, garanta que essa funcionalidade não será sobrescrita por nenhum código do CSS dos temas filho. É aqui onde o OOCSS ajuda bastante.
  • Você adicionará funcionalidades SEO à sua framework além daquelas que o WordPress provê, ou os usuários deverão adicionar isso através de plugins próprios para isso?
  • Você incluirá itens como sliders, galerias, imagens de plano de fundo, entre outras coisas, junto da sua framework ou elas deverão ser adicionas através dos temas filhos, caso necessário?

Essa lista pode crescer bastante, com o tempo, uma vez que suas necessidades e as daqueles para quem você desenvolve, mudam e/ou aumentam. Garanta que sua framework seja fácil de ser estendida desde o começo, e você será capaz de adicionar novas funcionalidades quando quiser.

Resumo

Desenvolver sua própria framework para temas é uma grande empreitada. É algo que economizará bastante do seu tempo a longo prazo, mas, para conseguir algo bem feito, demandará bastante tempo.

Ao parar para identificar quem usará sua framework e que características eles precisarão, você a tornará mais útil para si e para os outros usuários, e você a tornará maís fácil de estender e adaptar no futuro.
